# アプリの再設計

## 1. classからinterfaceへの移行（汎化より委譲）

### 背景
現在のアプリケーションは、クラスベースの設計を採用しており、特定の機能やデータ構造に依存しています。しかし、将来的な拡張性や保守性を考慮すると、クラスからインターフェースへの移行が望ましいと判断しました。これにより、異なる実装間での柔軟な切り替えが可能となり、コードの再利用性が向上します。

### 目的
- クラス依存の設計から脱却し、インターフェースを活用することで、コードの柔軟性を高める。
- 将来的な機能追加や変更に対応しやすい設計を実現する。
- コードの可読性と保守性を向上させる。

### 実装方針
1. **インターフェースの定義**
   - 現在のクラスの主要な機能を抽象化し、インターフェースとして定義する。
   - 例: `render`, `isHit`, `updatePosition` など。

2. **クラスのリファクタリング**
  - 既存のクラスをインターフェースに準拠するようにリファクタリングする。
  - 具体的な実装は、インターフェースを実装するクラスに委譲する。
  - インターフェースと会話することで、柔軟な実装の切り替えが可能となる。

### 期待される効果
- **柔軟性の向上**: インターフェースを介して異なる実装を容易に切り替えられるため、将来的な機能追加や変更に対応しやすくなる。
- **コードの再利用性**: インターフェースを利用することで、共通の機能を持つ複数のクラス間でコードを再利用できる。
- **保守性の向上**: クラスの依存関係が減少し、コードの可読性が向上するため、保守作業が容易になる。

### 理想

今は、H, H2, O にて、水(H2o)を表現しているが、将来的には、H, H2, O 以外の元素や分子を追加する可能性がある。そのため、現在のクラスベースの設計では、新しい元素や分子を追加する際に多くの変更が必要となる。インターフェースを導入することで、新しい元素や分子を追加する際の影響範囲を最小限に抑えることができる。

例えば、オゾン(O3)や二酸化炭素(CO2)などの新しい分子を追加する場合、インターフェースを利用することで、既存のコードに大きな変更を加えることなく、新しいクラスを実装することが可能となる。

描画や、衝突判定、位置更新などの共通の機能をインターフェースとして定義することで、新しい分子や元素が追加されても、既存のコードに影響を与えずに拡張できる。
衝突判定後にどうなるかは、インターフェースを通じて各分子や元素のクラスが独自に実装することができるため、柔軟な挙動を実現できる。

## 2. 高速化

### 背景
現在のアプリケーションは、描画や衝突判定などの処理において、パフォーマンスのボトルネックが存在しています。特に、衝突判定のアルゴリズムが非効率であり、多数の分子が存在する場合に処理速度が低下することが確認されています。

### 推測

1フレームの主要処理は src/simulator/MizuSimulator.ts:56 の renderFrame() です。

記号を置くと、n = H数, m = H2数, p = O数, q = H2o数, A = canvas面積(cw * ch) です。

結論

1フレームあたりの計算量は、衝突時の配列操作も含めると概ね

`O(A + n^2 + m + p*m + m^2 + q^2)`

同程度の個数を N と見るなら

`O(A + N^2)`

Canvas の fillRect / drawImage の内部ラスタライズを無視して「JS側のループ」だけを見るなら、支配的なのは H-H と O-H2 の総当たりで O(N^2) です。

内訳

| 処理 | 場所 | 計算量 |
|---|---|---|
| 画面クリア | `src/simulator/MizuSimulator.ts:57` | `O(A)` |
| H の描画 | `src/simulator/MizuSimulator.ts:114` | `O(n)` |
| H-H 衝突判定 | `src/simulator/MizuSimulator.ts:119` | `O(n^2)`、判定回数は最大 `n(n-1)/2` |
| H2 の描画 | `src/simulator/MizuSimulator.ts:138` | `O(m)` |
| O の描画 | `src/simulator/MizuSimulator.ts:144` | `O(p)` |
| O-H2 衝突判定 | `src/simulator/MizuSimulator.ts:148` | `O(p*m)` |
| O-H2 衝突時の `indexOf` / `splice` | `src/simulator/MizuSimulator.ts:154` | 最悪 `O(m^2 + p*m)` |
| H2o の描画・削除 | `src/simulator/MizuSimulator.ts:176` | 通常 `O(q)`、`splice` 多発の最悪 `O(q^2)` |
| バッファ転送 | `src/simulator/MizuSimulator.ts:65` | `O(A)` |


単体の衝突判定自体は、src/atoms/H.ts:51 と src/atoms/O.ts:52 のどちらも距離計算だけなので O(1) です。重くなる理由は判定式ではなく、全組み合わせを毎フレーム見ている点です。

### 方針

#### 高速化の方向性
衝突判定の計算量を O(N^2) から O(N log N) または O(N) に削減することを目指します。具体的には、空間分割アルゴリズム（例: クワッドツリー、グリッド分割）を導入し、衝突判定の対象を限定することで、計算量を削減します。また、描画処理においても、不要な描画を省略するための最適化を行い、パフォーマンスの向上を図ります。  

#### 計測可能

計測可能であることも条件として出します。フレームレートの可視化を出来るようにして、改善前後でのパフォーマンス比較を行います。
これは、例えばGetパラメータを使った場合に表示されるとか、デバッグ用のオーバーレイとして表示されるなど、ユーザーにとってもわかりやすい形で提供されることが望ましいです。
分子の数もコントロールできるようにして、負荷テストを行いやすくします。

#### 世界観

元のアプリの世界観を損なわないように、描画や挙動の変更は最小限に留めることを目指します。ユーザーが体験する見た目は、現行のアプリケーションと大きく変わらないように配慮します。

### 効果

- **パフォーマンスの向上**: 衝突判定の計算量を削減することで、フレームレートの向上が期待できる。
- **拡張性の向上**: 空間分割アルゴリズムを導入することで、将来的に分子の数が増加しても、パフォーマンスを維持しやすくなる。
- **好きな分子の追加**: 新しい分子や元素を追加する際にも、衝突判定のアルゴリズムが柔軟に対応できるようになる。
- **好きな結合の追加**: 新しい結合や反応を追加しやすくし、化学反応のシミュレーションの幅を広げることができる。
