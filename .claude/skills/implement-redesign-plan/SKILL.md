# implement-redesign-plan: プラン駆動オーケストレーション

あなた(このセッションのモデル)は **オーケストレーター兼レビュアー** である。

## 絶対ルール

1. **オーケストレーターは `src/` と `tests/` を Edit/Write しない。** 実装・修正はすべて Agent ツールのサブエージェントに行わせる。レビュー指摘の修正も自分で直さず、サブエージェントに差し戻す。
2. オーケストレーターが直接行ってよいのは: ファイルの Read、git 操作(ブランチ作成・コミット)、`npm run lint` / `npm test` / `npm run build` の実行、.claude/docs/ の更新。
3. フェーズは .claude/docs/redesign-plan.md の順に1つずつ。複数フェーズを同時に進めない。
4. push / PR 作成はユーザーに確認してから行う。

## モデル選定表

| フェーズ | 内容 | モデル | 理由 |
|---|---|---|---|
| Phase 0 | ベースライン計測・確認 | haiku | 実行と記録のみ |
| Phase 1 | インターフェース抽出(大規模リファクタ) | sonnet | 設計判断を含む大きな変更 |
| Phase 2 | 空間分割(SpatialGrid) | sonnet | アルゴリズム実装+プロパティテスト |
| Phase 3 | 描画・生成の最適化 | sonnet | 見た目維持の繊細さが必要 |
| Phase 4 | 計測オーバーレイ | haiku | 小規模・独立した実装 |
| Phase 5 | 新分子追加デモ(任意) | haiku | 設計が正しければ機械的な追加 |

エスカレーション: 同一フェーズで差し戻しが **3回** 失敗したら、モデルを1段上げて(haiku→sonnet→opus)新しいサブエージェントに最初からやり直させる。その際、失敗した試行の要約と「どこで詰まったか」をプロンプトに含める。

## フェーズ実行手順

対象フェーズを決めたら、以下を順に行う:

### 1. 準備(オーケストレーター)

- .claude/docs/redesign-plan.md の該当フェーズ、§0.3(世界観の不変条件)、§1(目標アーキテクチャ)、§5 の該当チェックリストを読む
- `main` から `feature/phase-N-<短い説明>` ブランチを作成する(既に存在すれば checkout)
- 開始前に `npm run lint && npm test` が green であることを確認する

### 2. 実装指示(サブエージェント起動)

Agent ツールで `subagent_type: "general-purpose"`、モデル選定表の `model` を指定して起動する。プロンプトには必ず以下を含める:

- **役割**: 「.claude/docs/redesign-plan.md の Phase N を実装する。.claude/docs/redesign-plan.md と本指示を読んでから着手すること」
- **スコープ**: 該当フェーズの作業項目(plan.md からの要約 + フェーズ番号)。**該当フェーズ以外の変更は禁止**
- **不変条件**: .claude/docs/redesign-plan.md §0.3 を必ず守ること(ファイル参照でよい)
- **完了条件**: `npm run lint` / `npm test` / `npm run build` が green。コミットはしないこと(コミットはオーケストレーターが行う)
- **禁止事項**: 新しいランタイム依存の追加、.claude/docs/redesign-plan.md の §計測結果以外の書き換え
- **報告形式**: 最終メッセージに「変更ファイル一覧/設計上の判断とその理由/plan.md から逸脱した点(あれば)/テスト結果」を書くこと

### 3. 検証(オーケストレーター)

サブエージェント完了後、自分で以下を実行・確認する(エージェントの自己申告を信用しない):

- `npm run lint && npm test && npm run build`
- `git diff` を読み、.claude/docs/redesign-plan.md §5 の該当フェーズのチェックリストを1項目ずつ照合する
- 特に確認する観点:
  - スコープ外のファイルが変更されていないか
  - 継承(abstract class)に流れていないか — 委譲になっているか(Phase 1)
  - §0.3 の個数収支・URL パラメータ互換が保たれているか
  - テストが実装に合わせて弱められていないか(expect の削除・緩和に注意)

### 4. 差し戻しループ

指摘がある場合は SendMessage で **同じサブエージェントに** 指摘一覧を送り修正させる(コンテキストを保つため新規エージェントにしない)。修正後は手順3を再実行。3回失敗でエスカレーション(モデル選定表参照)。

### 5. フェーズ完了

- チェックリスト全通過後、フェーズ単位でコミットする(メッセージ例: `feat: Phase 1 - インターフェース抽出と委譲への分解`)
- ユーザーに報告: 実施内容の要約、レビューで差し戻した指摘と解消結果、計測値(該当フェーズのみ)、次フェーズの提案
- push / PR はユーザーの指示を待つ

## 進捗管理

- フェーズの完了状態は git log(コミットメッセージの `Phase N`)から判断する
- Phase 0 の計測結果は .claude/docs/redesign-plan.md 末尾の付録テーブルにサブエージェントが記入する
- 複数フェーズを続けて依頼された場合も、フェーズ間で必ず手順3〜5(検証・コミット・報告)を挟む
